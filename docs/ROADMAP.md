# ROADMAP

## Goal

CLI から小説プロジェクトを作成し、長編小説を
`設計 -> 分解 -> 執筆 -> 検査 -> 再計画 -> 改稿 -> 公開用成果物出力`
まで自律的に回せる制作システムへ育てる。

最終目標は「章を順番に書けること」ではなく、**長い作品を壊さずに最後まで運べること**である。
この文書は中長期の到達点と milestone の見取り図を保ち、判断基準は `docs/DEVELOPMENT_GUIDE.md` に寄せる。

## Product Direction

このソフトウェアは、単発で大きな本文を出し切ることを最大化する方向は目指さない。
目指すのは、設計、分解、執筆、検査、再計画、改稿を分担しながら連携できる production system である。
長い作品を扱えるかどうかは、1 回の生成を長くすることではなく、各段階を壊さずにつないで運べるかで決まる。
そのために、次の 5 能力を備えた長編運用基盤を育てる。

1. **設計能力**
   作品全体の premise、登場人物、世界設定、伏線、終着点を保持する。
2. **分解能力**
   act -> chapter -> scene へ落とし、各段階に目的と制約を持たせる。
3. **記憶能力**
   長編の途中でも、既出事実・未回収要素・人物変化を再利用できる。
4. **検査能力**
   continuity だけでなく、進行停滞、伏線未回収、感情線の欠落を検知する。
5. **再計画能力**
   問題発見時に、章単位 rerun だけでなく以降の計画を更新できる。

## Current State

長編制作の基礎層はすでにあり、設計、分解、記憶、評価、再計画の前提は揃っている。
いまの焦点は、その土台の上で自律性と成果物の厚みをどこまで伸ばすかである。

## Gap To Goal

実装済みの土台がある一方で、長編を最後まで運ぶには次の能力をさらに伸ばす必要がある。

### 1. 自律判断の完成度

人手に依存しすぎず、続行・改稿・再実行・再計画・停止を安定して切り替える必要がある。

### 2. 成果物束の実用性

完成原稿だけでなく、編集・レビュー・再生成に使える形へ成果物を広げる必要がある。

### 3. 長期記憶と参照性

長編の途中で蓄積される事実、伏線、関係、状態を再利用しやすくする必要がある。

### 4. 長編評価の深さ

中盤停滞、役割重複、感情線の停滞、伏線回収漏れのような長編特有の崩れを、より明確に捉える必要がある。

## Completed Foundations

- 作品設計、章分解、長期記憶、評価、再計画の基本層は導入済みである。
- 章生成の入力整理と、後続工程が参照するための基礎的な連携はできている。
- 長編の中盤から終盤に向けた運用の土台はある。

## Remaining Roadmap

### M68–M70. 自律継続ループを固定する

目的:
`TASKS.md` の queue 補充、停止記録、autonomy level と workflow の接続を整理し、
「次に何をやるか」で止まりにくい運転ルールを固定する。

完了条件:

- `Ready` が空のときの最小子 task 補充ルールを説明できる
- `BLOCKED.md` の停止記録粒度が一定になる
- `manual` / `assist` / `auto` の差を workflow として説明できる

### M71–M75. Story State の共有断面を広げる

目的:
resume / rerun / comparison / publish で使う最小 story-state を揃え、
長編途中でも文脈を再利用しやすくする。

完了条件:

- 再開判断に必要な story-state が各 saved artifact で揃う
- view 間の story-state 表示がより一貫する

### M76–M80. 次アクション自動判断を強化する

目的:
continue / revise / rerun / stop_for_review の判断を、
saved decision と project autonomy policy からより説明しやすくする。

完了条件:

- action 選択理由を artifact から説明できる
- `resume-project` と rerun 判断の一貫性が上がる

### M81–M86. 長編安定運用を強化する

目的:
chapter 間依存、thread 回収、replan を強化し、中盤以降で崩れにくくする。

完了条件:

- 長編の再計画で持ち直しやすくなる
- 中盤以降の崩れを早めに検知しやすくなる

### M87–M90. 選抜・公開ワークフローを完成させる

目的:
生成、比較、採用、公開準備までの実務フローを閉じる。

完了条件:

- best run の選抜根拠が整理される
- publish bundle が公開候補判断に使いやすくなる

## Sequencing Rationale

- まず M68〜M70 で「止まりにくい運転ルール」を固定する
- 次に M71〜M75 と M76〜M80 で story state と next action decision を強化する
- その後 M81〜M86 で長編安定運用を厚くし、M87〜M90 で選抜・公開まで閉じる

## Roadmap Notes

- 詳細な判断基準は `docs/DEVELOPMENT_GUIDE.md` に寄せる
